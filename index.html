Overview

  • Distribution per Run (100k): Bit-bounded sampling by field size. Example: a 32-bit timestamp ~40,000 randomized cases; a 16-bit field ~20,000. Ratios informed by prior research.
  • Targeting: Tests iterate fields/commands sequentially; direct single-field/command targeting isn‚Äôt supported.
  • Uniqueness Evidence: One full run executed; see log.txt for unique entries (per protocol notes below).

RTP

  • Distribution: as above (e.g., 32-bit timestamp ~40k).
  • Targeting: sequential only.
  • Uniqueness: evidence in log.txt.
  • Connectivity Loss: connectionless; diagnose via device logs or Wireshark‚Äîprogram cannot infer loss itself.
  • Packet Recording: logs contain the full packet content required to reproduce frames; Wireshark‚Äôs leading link/endpoint headers are auxiliary.

RTCP

  • Distribution/Targeting: same as RTP.
  • Uniqueness: full-run evidence in log.txt.
  • Connectivity Loss: connectionless; use device logs / Wireshark.
  • Packet Recording: same rationale as RTP.

RTSP

  • Distribution: cases allocated by command; each run shuffles/adds/removes/mutates commands.
  • Targeting: per-command isolation not supported; tested in order.
  • Uniqueness: see log.txt.
  • Connectivity Loss: connection-oriented; if the connection drops, the program stops‚Äîuse timestamps to locate the triggering packet.
  • Recording: logs focus on command payloads; generic link headers aren‚Äôt required for reproduction.

TLS

  • Distribution: 50k fixed-header + 50k fully randomized.
  • Targeting: no direct jump to a specific field/strategy.
  • Uniqueness: a full 100k run wasn‚Äôt completed due to time; randomization mirrors RTP/RTCP.
  • Connectivity Loss: TLS over TCP‚Äîany disconnection interrupts transmission.
  • Logging/Wireshark:
    • Fixed-header phase: Wireshark identifies TLS; payloads encrypted‚Äîcorrelate with timestamps.
    • Full-fuzz phase: mutated headers may appear as SSL; without encryption, payload bytes remain visible.
