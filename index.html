TP Protocol Fuzzing Suite

High-volume fuzz testing for RTP, RTCP, RTSP, and TLS ‚Äî up to 100,000 cases per run, with reproducible logs.

RTPRTCPRTSPTLS Randomized CasesLog-based Evidence

Overview

  • Distribution per Run (100k cases): For bit-bounded fields, the tool samples by field size. Example: a 32-bit timestamp is randomized ~40,000 cases; 16-bit fields ~20,000. Sampling ratios are informed by related research.
  • Field/Command Targeting: The engine iterates fields/commands in sequence; direct jumping to test a single field/command is not supported.
  • Uniqueness Evidence: A full end-to-end run was executed; see log.txt for 100k unique entries (per protocol notes below).

RTP

  • Distribution: Bit-size guided sampling as above (e.g., 32-bit timestamp ~40k cases).
  • Targeting: Sequential; cannot isolate a specific field only.
  • Uniqueness Proof: Provided via log.txt from a complete run.
  • Connectivity Loss: RTP is connectionless; loss must be inferred from device logs or Wireshark captures. The program cannot determine loss by itself.
  • Packet Recording: Logged packets are complete. Wireshark‚Äôs leading link/endpoint headers are auxiliary; reconstruction is possible from log.txt content alone.

RTCP

  • Distribution: Same method as RTP.
  • Targeting: Same as RTP (sequential only).
  • Uniqueness Proof: Full-run evidence in log.txt.
  • Connectivity Loss: Connectionless; diagnose via device logs/Wireshark.
  • Packet Recording: Same rationale as RTP.

RTSP

  • Distribution: Test counts are allocated by command type; each run shuffles, adds/removes, and mutates commands randomly.
  • Targeting: Per-command isolation isn‚Äôt supported; commands are tested in order.
  • Uniqueness Proof: Full-run evidence in log.txt.
  • Connectivity Loss: RTSP is connection-oriented; if the connection drops during transmission, the program stops. Use log.txt to locate the specific triggering packet.
  • Packet Recording: Logs focus on command payloads. Link/basic headers seen first in Wireshark are generic; reproduction doesn‚Äôt require them.

TLS

  • Distribution: 50k with fixed header + 50k fully randomized.
  • Targeting: No direct jump to a given field/strategy.
  • Uniqueness Proof: A full 100k run was not completed due to time; the randomization approach mirrors RTP/RTCP.
  • Connectivity Loss: TLS over TCP‚Äîany disconnection interrupts transmission.
  • Logging/Wireshark Notes:
    • During fixed-header phase, Wireshark identifies TLS, but payloads are encrypted‚Äîuse timestamps to correlate issues.
    • During full-fuzz phase (header mutated), Wireshark may label as SSL; without encryption you can inspect packet contents.
